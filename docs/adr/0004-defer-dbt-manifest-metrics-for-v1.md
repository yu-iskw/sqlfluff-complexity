# ADR 0004: Defer dbt manifest metrics for v1

- **Status:** Accepted
- **Date:** 2026-05-01
- **Deciders:** Maintainers

## Context

The primary users are dbt analytics engineering and data platform teams, so dbt compatibility matters from the first release. However, dbt-aware complexity can mean several different things: linting source SQL through SQLFluff's dbt templater, analyzing compiled SQL, reading `manifest.json`, or computing DAG-level fan-in and fan-out metrics.

Adding manifest analysis in v1 would introduce a larger scope, optional dependency questions, and a second source of complexity semantics before the core SQL metrics are proven.

## Decision

v1 will support dbt projects through SQLFluff configuration and templating, but it will not parse dbt `manifest.json` or compute DAG-level metrics.

The stable invariant is that v1 measures SQLFluff-parsed SQL, not full dbt project complexity. dbt manifest and compiled-SQL analysis may be added later as an optional extension, but the v1 architecture should avoid a hard dbt dependency and keep SQL segment metrics useful for dbt source models.

## Consequences

- The first release stays focused on SQLFluff plugin behavior, core SQL metrics, and reporting.
- Users can run the plugin in dbt projects that already use SQLFluff's dbt templater.
- Complexity generated by macros or visible only in compiled SQL may be undercounted in v1.
- Future dbt-aware work needs clear documentation because it may change the meaning of model-level complexity.

## Alternatives considered

- **Require dbt and parse `manifest.json` in v1:** Richer model graph insight, but significantly larger scope and dependency surface.
- **Analyze only compiled SQL:** Captures macro-expanded complexity, but makes local source-level feedback harder to explain and requires a dbt compile workflow.
- **Ignore dbt compatibility initially:** Smaller implementation, but misses the primary audience and adoption path.

## Trade-offs

We accept known undercounting for macro-expanded and DAG-level complexity in order to ship a smaller, maintainable v1. The limitation must be documented so users understand that v1 measures SQLFluff-parsed SQL, not full dbt project complexity.
